## Chat integration (the important part)
The right column of `.stage` is the live chat. Reproduce the prototype's `.talk` column
chrome — the `.talk-top` status bar (pulsing crimson dot + "Bussey & Bussey" + "online"),
the rotating `.ask` line at the top of the thread, the `.begin` cue ("↓ Begin here.
Speak with our Automated Intelligence."), and the `.composer` with the crimson square
↑ send button — but the MESSAGE THREAD and INPUT must be the real widget, not static divs.

Approach:
- In `index.njk`, build the `.talk` column with: `.talk-top` (static chrome), the rotating
  `.ask` element (id `bb-tagline`, keep the existing rotation JS + 6 lines), then the
  `#bb-chat-inline` mount where the real thread+composer render, then the `.lock` line.
- The `.begin` cue line sits just above the composer. Since the composer is generated by
  the widget JS in inline mode, EITHER (a) move the `.begin` line into the widget's inline
  DOM (render it just above the textarea), or (b) place it in index.njk and ensure it sits
  visually directly above the widget's input. Either is fine; it must read as one unit:
  begin-cue → input → send.
- Restyle the widget's inline CSS (`.bb-chat-panel--inline` and the message/input/button
  rules) to match the prototype EXACTLY: assistant bubble `#f4f4f1` with radius
  `4px 14px 14px 14px`; user bubble `--accent` white with radius `14px 14px 4px 14px`;
  thread padding 30px 26px, gap 16px; input 1.5px `#d8d8d2` border, radius 11px, 50px tall,
  focus border `--ink`; send button 50px crimson square, radius 11px, ↑ glyph, hover
  `--accent-deep #9e0816`; blank placeholder.
- The real greeting still comes from the worker (INITIAL_GREETING) — that's fine, it renders
  in the thread. The rotating `.ask` line is SEPARATE chrome above the thread, not a message.
- Keep the corner-bubble fallback for all other pages unchanged.
